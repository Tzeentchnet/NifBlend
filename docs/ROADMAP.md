# NifBlend Roadmap

> Contributor-facing source of truth for what NifBlend is building, in what order, and how we'll know it works. Tick checkboxes as steps land. Record what *shipped* in [`CHANGELOG.md`](CHANGELOG.md).

## TL;DR

NifBlend is a greenfield Blender 5.0+ extension for importing/exporting Bethesda NIF and KF files (Morrowind → Fallout 76 → **Starfield import in v1.1**; Starfield export deferred to v2). Architecture: an **in-house generated schema layer** (`nifblend/format/generated/`, emitted from a pinned snapshot of `niftools/nifxml`, GPL-3.0) plus a **clean-room, numpy-vectorized Blender bridge** for speed and clarity. Ships in slices: Skyrim SE static mesh first, then materials, armature/skin, KF, then per-game expansion, then Starfield import.

## Architecture overview

```
nifblend/
├── manifest/blender_manifest.toml   # Blender 5.x extension manifest (numpy wheel dep)
├── __init__.py                      # registration, operators, menu wiring
├── schema/nif.xml                   # pinned snapshot of niftools/nifxml (GPL-3.0); single source of truth
├── format/                          # game-specific dispatch + generated schema layer
│   ├── base.py / primitives.py / versions.py   # hand-written runtime substrate
│   └── generated/                   # blocks.py, structs.py, enums.py, bitfields.py — emitted by tools/codegen
├── vendor/nifgen/                   # reference-only: empty; `blender_niftools_addon`'s nifgen may be consulted for design parity
├── io/                              # buffered + numpy-vectorized binary I/O
│   ├── reader.py / writer.py
│   ├── header.py
│   └── block_table.py
├── format/                          # (continued) game-specific dispatch
│   ├── versions.py                  # GameProfile enum from (version, user, bs_version)
│   ├── bstrishape.py
│   ├── bsgeometry.py
│   └── starfield_mesh.py            # v1.1 — reserved
├── bridge/                          # Blender ↔ NIF translation (clean-room)
│   ├── mesh_in.py / mesh_out.py
│   ├── material_in.py / material_out.py
│   ├── armature_in.py / armature_out.py
│   └── animation_in.py
├── ops/                             # Blender operators
│   ├── import_nif.py / export_nif.py
│   ├── import_kf.py
│   ├── import_batch.py / export_batch.py
│   └── preview_lod.py
├── preferences.py                   # Data-folder roots per game, defaults
└── tests/                           # pytest suite + per-game golden NIFs

tools/codegen/                       # in-house schema → Python emitter
├── parser.py                        # nif.xml → AST
├── cond_compiler.py                 # schema DSL → Python expressions
├── whitelist.py                     # seed types; closure walked automatically
├── emit.py                          # writes blocks/structs/enums/bitfields
└── __main__.py                      # `python -m tools.codegen [--check]`
```

### Phase 2 — MVP: Skyrim SE static mesh (BSTriShape) round-trip

- [x] 5. `io/reader.py` + `io/writer.py`: buffered file readers/writers wrapping the scalar/bulk primitives in [`nifblend/format/primitives.py`](../nifblend/format/primitives.py); add any missing array readers (vec3, half-float vec2, packed BSVertexData) discovered while wiring BSTriShape
- [x] 6. `io/header.py` + `io/block_table.py`: parse the NIF header (version/user/bs_version, block-type table, block sizes, string table, group sizes), drive a per-block read loop using the generated `Block.read(stream, ctx)` classmethods, expose a flat `BlockTable` with raw u32 refs preserved; inverse for write
- [x] 7. `format/versions.py` (extend): add `GameProfile` enum with detection from `(version, user_version, bs_version)`; cover Skyrim LE/SE first
- [x] 8. `bridge/mesh_in.py`: BSTriShape → `bpy.data.meshes.new` via `foreach_set` for positions/normals/UVs/colors/tangents. Resolves block refs and string-table indices into Blender semantics. **First-class BSTriShape** — preserved on round-trip, no NiTriShape conversion
- [x] 9. `bridge/mesh_out.py`: inverse — `foreach_get`, build BSVertexData with appropriate flags, repack indices, allocate block-table slots and string-table entries
- [x] 10. Wire [`ops/import_nif.py`](../nifblend/ops/import_nif.py) + [`ops/export_nif.py`](../nifblend/ops/export_nif.py) (currently stubs) to the bridge, then add round-trip tests: import → export → re-import; assert vertex/index/UV equality within float epsilon for ≥3 sample SSE meshes. Use failures here to drive any whitelist/codegen widening (step 2f) rather than speculating up front. Operator wiring landed alongside Phase 2.8/2.9; round-trip coverage in [`nifblend/tests/test_round_trip.py`](../nifblend/tests/test_round_trip.py) parameterises three distinct SSE meshes (triangle, indexed quad, tetrahedron) plus a multi-block NIF and asserts topology + positions are bit-exact, UVs within `1e-3` (half-float), and normals/colors within `1/127` (u8 quantisation).

### Phase 3 — Materials
*Depends on Phase 2.*

- [x] 11. `bridge/material_in.py`: BSLightingShaderProperty + BSEffectShaderProperty → Principled BSDF group with diffuse/normal/specular/glow texture nodes; resolve texture paths against pref-configured Data roots (case-insensitive on Windows). Pure decoder lives in [`bslighting_to_material_data`](../nifblend/bridge/material_in.py) / [`bseffect_to_material_data`](../nifblend/bridge/material_in.py); Blender wrapper [`material_data_to_blender`](../nifblend/bridge/material_in.py) takes an injectable `bpy` and a `resolve_texture` callback so disk-path resolution stays a preferences concern.
- [x] 12. `bridge/material_out.py`: inverse, reading from "NifBlend Material" node group output to preserve shader flags. Pure builders [`material_data_to_bslighting`](../nifblend/bridge/material_out.py) / [`material_data_to_bseffect`](../nifblend/bridge/material_out.py) plus [`build_texture_set`](../nifblend/bridge/material_out.py) and [`build_alpha_property`](../nifblend/bridge/material_out.py) factor block construction out of the operator; [`material_data_from_blender`](../nifblend/bridge/material_out.py) walks the predictable Principled-BSDF + Image/NormalMap graph the import side builds. Cross-block refs (`name_index`, `texture_set_ref`) stay the operator's responsibility.
- [x] 13. Material flag preservation via typed `PropertyGroup` registered on `Material` (round-trip even when Blender can't represent the flag). [`nifblend/bridge/material_props.py`](../nifblend/bridge/material_props.py) defines `NifBlendMaterialProperties` (origin, raw shader_type, both 32-bit shader_flags words, glossiness, specular_strength, specular_color, alpha_threshold, uv_offset/scale, emissive_multiple) plus a `NifBlendMaterialTextureSlot` `CollectionProperty` for any slots beyond diffuse/normal/glow that the Principled-BSDF graph wires natively. `apply_material_data_to_props` / `read_material_data_from_props` are duck-typed (work against the real `PointerProperty` and against `SimpleNamespace` test fakes) and are called from `material_data_to_blender` / `material_data_from_blender` so the Principled graph wins for fields Blender can represent and the PropertyGroup wins for everything else. Registered from [`nifblend/__init__.py`](../nifblend/__init__.py) alongside the operators.

### Phase 4 — Armature & skinning
*Depends on Phase 2; parallel with Phase 3.*

- [x] 14. `bridge/armature_in.py`: NiNode tree → Armature; preserve full 4×4 bind matrix on each bone via typed `PropertyGroup` (bone roll alone is lossy). Pure decoder [`ninode_tree_to_armature_data`](../nifblend/bridge/armature_in.py) walks the `NiNode` children graph (skipping non-NiNode refs and invalid u32 indices), composes each node's `translation` + `Matrix33` rotation + uniform `scale` into a 4×4 local bind matrix, pre-composes world matrices through `parent.world_matrix @ local`, and topologically orders bones parent-before-child. Blender wrapper [`armature_data_to_blender`](../nifblend/bridge/armature_in.py) creates an Armature object, lays out one EditBone per `BoneData` (head = world translation; tail = head + length × world Y-axis; length = distance to first child or `DEFAULT_BONE_LENGTH`), then stamps the lossless local 4×4 onto each `bpy.types.Bone.nifblend.bind_matrix` PropertyGroup ([`nifblend/bridge/armature_props.py`](../nifblend/bridge/armature_props.py)) so the `(head, tail, roll)` reduction Blender forces on us is always recoverable on export.
- [x] 15. NiSkinInstance / NiSkinPartition / BSSkin::Instance → vertex groups, vectorized via numpy (no per-vertex Python loop). Sparse [`SkinData`](../nifblend/bridge/skin_in.py) dataclass (positional `bone_names` palette + parallel `vertex_indices` / `bone_indices` / `weights` numpy arrays, zero weights pre-filtered) decoded by [`niskin_to_skin_data`](../nifblend/bridge/skin_in.py) for the classic `NiSkinInstance` + `NiSkinData` layout (also `BSDismemberSkinInstance` via duck-typing) and [`bstrishape_skin_to_skin_data`](../nifblend/bridge/skin_in.py) for the SSE / FO4 per-vertex (`BSVertexDataSSE.bone_indices[4]` / `bone_weights[4]`) layout. Both decoders pre-size destination arrays and use `numpy.fromiter` / vectorised flattening — no per-vertex Python loop walks the NIF block lists. [`apply_skin_to_object`](../nifblend/bridge/skin_in.py) creates one Blender vertex group per palette bone and bucket-batches `vertex_group.add` calls per `(bone, unique_weight)` pair (one Blender API call per influence-class, not per vertex). `NiSkinPartition` is a GPU-skinning hint, not the source of truth for weights, so we read but do not consume it on import; it is regenerated on export in step 16.
- [x] 16. `bridge/armature_out.py`: rebuild skin partitions, optimize splits per game limits (SLE/SSE 80 bones/partition, FO4 different). Pure (no `bpy`) module exposes a `BoneLimits` dataclass + per-`GameProfile` `bone_limits_for()` lookup (Morrowind 4 bones / 4 weights, Oblivion / FO3-NV 18 / 4, Skyrim LE+SE 80 / 4, FO4 80 / 8, FO76 100 / 8, UNKNOWN → conservative SSE default), a `PartitionBuild` dataclass holding the per-partition `vertex_map` / local triangles / local bone palette / `(N, K)` `bone_indices` + `vertex_weights` arrays, and three top-level entry points: `build_skin_partitions(skin, triangles, num_vertices, *, limits)` greedy first-fit packs triangles into partitions whose union-bone-set stays within `limits.max_bones_per_partition` (per-vertex influences pre-pruned to top-K and renormalised; over-sized triangles raise `ValueError`); `skin_partitions_to_block(parts, *, num_weights_per_vertex)` wraps the builds in a `NiSkinPartition` (LE-shaped — SE/FO4 callers patch `vertex_desc` / `vertex_data` / `data_size` / `vertex_size` from the corresponding `BSTriShape` themselves so the partition builder stays game-neutral); `build_ni_skin_data(skin)` is the inverse of `niskin_to_skin_data` (per-bone `BoneVertData` lists, palette slots preserved even when empty, identity skin transform); `build_ni_skin_instance(skin, *, data_ref, partition_ref, skeleton_root_ref, bone_block_refs, dismember=False)` emits an `NiSkinInstance` (or `BSDismemberSkinInstance` shell) with cross-block refs left to the operator. Body-part dismember partitions stay deferred to step 17.
- [x] 17. BSDismemberSkinInstance partition support: store partition ID per vertex group via typed `PropertyGroup`. [`nifblend/bridge/skin_props.py`](../nifblend/bridge/skin_props.py) registers `NifBlendVertexGroupProperties` (`is_partition` flag plus the verbatim u16 `part_flag` / `body_part` pair from the source `BodyPartList`) on `bpy.types.VertexGroup` so dismember partitions can ride alongside the bone-weight groups produced by [`apply_skin_to_object`](../nifblend/bridge/skin_in.py) without colliding. Duck-typed helpers `apply_partition_to_props` / `read_partition_from_props` / `clear_partition_on_props` work against both the real `PointerProperty` and `SimpleNamespace` test fakes. End-to-end converters `bodypart_lists_to_vertex_groups(parts, obj, *, name_prefix="BP")` (creates one `BP_{i}_{body_part}`-named group per partition and stamps the pair onto each) and `vertex_groups_to_bodypart_lists(groups)` (harvests partition-marked groups back into a `list[BodyPartList]`, silently skipping bone-weight groups) give the operator a single-call round-trip path. Raw u16 ints are preserved so unknown `BSDismemberBodyPartType` values (modder-defined parts past the codegen enum) survive untouched. Wired through `nifblend/__init__.py`'s `register()` / `unregister()` alongside `material_props` and `armature_props`.

### Phase 5 — KF animation import
*Depends on Phase 4.*

- [x] 18. KF file = standalone NIF with NiControllerSequence root; reuse Phase 1–2 reader. New module [`nifblend/io/kf.py`](../nifblend/io/kf.py) exposes `is_kf_file(table)` (true iff every footer-root index resolves to a `NiControllerSequence`; rejects empty roots, out-of-range / negative indices, and mixed `NiNode`+sequence roots), `kf_root_sequences(table)` (returns the rooted sequences in declaration order, raises `ValueError` for non-KF tables), and `iter_controller_sequences(table)` (yields every `NiControllerSequence` in block order — also surfaces sequences referenced indirectly from a `NiControllerManager` inside a regular `.nif`). The Phase 2.6 [`read_nif`](../nifblend/io/block_table.py) / [`write_nif`](../nifblend/io/block_table.py) pipeline already round-trips KF-shaped tables byte-faithfully; this step is the file-shape-detection / iteration surface the bridge layer (Phase 5.19) and the `import_kf` operator (5.21) will call. Test suite [`nifblend/tests/test_kf_io.py`](../nifblend/tests/test_kf_io.py) (12 tests) covers detection (empty / NiNode root / sequence root / out-of-range root / negative root / mixed roots), root-sequence harvesting (declaration order, non-KF rejection), `iter_controller_sequences` over a regular `.nif` carrying an embedded sequence, and end-to-end **single-sequence and multi-sequence KF round-trips** through `write_nif` → `read_nif` asserting `name` / `weight` / `frequency` / `cycle_type` / `start_time` / `stop_time` / `text_keys` / `manager` / `num_anim_note_arrays` survive. Per-keyframe decoding (`Key` / `QuatKey` template-argument fields) is deferred to step 19, so the test fixtures use `num_controlled_blocks = 0` to keep this step focused on file-shape parity.
- [x] 19. `bridge/animation_in.py`: NiControllerSequence → Action; for each NiTransformInterpolator → fcurves. **Vectorized keyframe insertion** — pre-allocate `keyframe_points.add(n)` then `foreach_set("co", flat)` (10–100× faster than per-frame `keyframe_insert`). Pure decoder [`controller_sequence_to_animation_data`](../nifblend/bridge/animation_in.py) walks each `ControlledBlock`, resolves `interpolator → data` through `NiTransformInterpolator → NiTransformData`, and packs `Key` / `QuatKey` lists into flat `float32` arrays where column 0 is `time * fps` (NIF seconds → Blender frames; default 30). `BoneTrack` carries per-bone `(N, 4)` translation, `(N, 5)` `(frame, w, x, y, z)` quaternion rotation, three per-axis `(N, 2)` arrays for `XYZ_ROTATION_KEY` Euler streams, `(N, 2)` uniform scale, plus per-channel raw `KeyType` interpolation codes. Blender wrapper [`animation_data_to_blender`](../nifblend/bridge/animation_in.py) creates a `bpy.types.Action`, emits one fcurve per active channel index (`pose.bones["X"].location[0..2]`, `.rotation_quaternion[0..3]` or `.rotation_euler[0..2]`, `.scale[0..2]`), and bulk-loads each stream via `keyframe_points.add(n)` + `foreach_set("co", flat)` + `foreach_set("interpolation", codes)`. NIF Quaternion `(w, x, y, z)` matches Blender's quaternion order (no swizzle); NIF's uniform scale fans out to all three Blender scale indices on import. `Key.value is None` rows are dropped so the bridge stays robust until the codegen `#T#` template gap on `Key` / `QuatKey` (step 2f) closes — once it does, real `.kf` files decode end-to-end with no further bridge changes.
- [x] 20. Quaternion → Euler / native quaternion rotation mode toggle in import operator. [`nifblend/bridge/animation_in.py`](../nifblend/bridge/animation_in.py) exposes `ROTATION_MODES = ("QUATERNION", "EULER")`, [`quaternion_stream_to_euler_streams`](../nifblend/bridge/animation_in.py) (vectorised quat → intrinsic XYZ Euler conversion preserving the `(frame, value)` time column), and [`apply_rotation_mode_to_armature`](../nifblend/bridge/animation_in.py) (sets `pose.bones[<name>].rotation_mode = 'XYZ'` on every animated bone when EULER is selected so Blender's pose evaluator picks up the right channel). `animation_data_to_blender(..., rotation_mode=...)` routes the per-bone quaternion stream through the converter and emits three `rotation_euler[0..2]` fcurves instead of four `rotation_quaternion[0..3]` ones; QUATERNION stays the default lossless path.
- [x] 21. [`ops/import_kf.py`](../nifblend/ops/import_kf.py) operator + retargeting onto a selected armature. [`NIFBLEND_OT_import_kf`](../nifblend/ops/import_kf.py) (`bl_idname="nifblend.import_kf"`) extends `ImportHelper` with a `*.kf` filter, the Phase 5.20 `rotation_mode` enum, and a dynamic `target_armature` enum populated from every `ARMATURE` object in the scene (sentinel `<no armature in scene>` entry when empty). `execute` calls [`read_nif`](../nifblend/io/block_table.py), gates on [`is_kf_file`](../nifblend/io/kf.py), resolves the target armature (explicit pick wins, else `context.active_object` if it's an armature), then for each footer-root sequence decodes through [`controller_sequence_to_animation_data`](../nifblend/bridge/animation_in.py), materialises an Action via [`animation_data_to_blender`](../nifblend/bridge/animation_in.py), applies [`apply_rotation_mode_to_armature`](../nifblend/bridge/animation_in.py), and assigns the action to `armature.animation_data.action` (creating the AnimData if needed). Bone tracks missing from the target armature are surfaced as a single `WARNING` report (Blender keeps the fcurve data path live, so the action starts driving the moment the bone is added). Wired into `File → Import` via [`nifblend/__init__.py`](../nifblend/__init__.py).

### Phase 6 — Per-game expansion
*Steps within phase parallel after stub adapter exists.*

- [x] 22. Oblivion (NIF 20.0.0.5, NiTriStrips path) — strips→triangles conversion in `mesh_in`, no BSTriShape. Pure decoder [`strips_to_triangles`](../nifblend/bridge/mesh_in.py) walks each strip with the standard D3D alternating-winding rule (`(a,b,c)`, `(b,d,c)`, `(c,d,e)` …) and drops degenerate triples; [`nitrishape_to_mesh_data`](../nifblend/bridge/mesh_in.py) and [`nitristrips_to_mesh_data`](../nifblend/bridge/mesh_in.py) pack the on-data vertex / normal / colour / UV-set-0 streams into the same `MeshData` container the BSTriShape path produces, so [`ops/import_nif.py`](../nifblend/ops/import_nif.py) only needs an `isinstance` branch per shape type to dispatch a Morrowind / Oblivion / FO3-NV NIF into Blender. Prerequisite codegen widening landed in the same step: [`tools/codegen/cond_compiler.py`](../tools/codegen/cond_compiler.py)'s `compile_width` and [`tools/codegen/emit.py`](../tools/codegen/emit.py)'s width-aware array reader/writer fix the pre-existing under-read of `NiTriStripsData.points` (and `SkinPartition.points` / `.strips` / `.vertex_weights` / `.bone_indices`) by emitting `[read_array_u16(stream, int(self.strip_lengths[__i])) for __i in range(int(self.num_strips))]` for jagged sibling-array widths and `[[…] for _ in range(N)]` 2D grids for scalar / literal widths. Container array-fieldness is threaded through `_EmitContext.container_arrays`, which is populated for the duration of `_emit_struct` / `_emit_block` so the heuristic distinguishes `width="Strip Lengths"` (per-row) from `width="Num Weights Per Vertex"` (scalar). Test suites [`nifblend/tests/test_mesh_in_legacy.py`](../nifblend/tests/test_mesh_in_legacy.py) (12 tests covering strip decoding edge cases + both classic-shape decoders) and [`nifblend/tests/test_oblivion_strips.py`](../nifblend/tests/test_oblivion_strips.py) (2 tests round-tripping an Oblivion-shaped `NiTriStripsData` through `.write()` → `.read()` then through the bridge) gate the change.
- [x] 23. Fallout 3 / NV (20.2.0.7 user 11/34) — BSShaderPPLightingProperty material path. Pure decoder [`bsshaderpplighting_to_material_data`](../nifblend/bridge/material_in.py) handles the pre-Skyrim FO3/NV lighting shader: u16 `BSShaderFlags` `flags` word, single 32-bit `shader_flags` (no second word in this layout but routed through the same PropertyGroup slot the SSE path uses), `environment_map_scale` / `texture_clamp_mode`, refraction (`bs_version > 14`) and parallax (`> 24`) scalars, plus a six-slot `BSShaderTextureSet` walked through `_resolve_texture_set` (slot order matches the first six of `TEXTURE_SLOT_NAMES`: diffuse / normal / glow / height / environment / env_mask). The inline emissive `Color4`'s alpha channel doubles as `emissive_multiple` (FO3/NV has no separate field). [`material_in.import_material`](../nifblend/bridge/material_in.py) gains an `isinstance(BSShaderPPLightingProperty)` branch so a single call decodes + materialises against the Principled-BSDF graph the SSE path already uses. Inverse [`material_data_to_bsshaderpplighting`](../nifblend/bridge/material_out.py) + companion [`build_pp_texture_set(data, *, num_slots=6)`](../nifblend/bridge/material_out.py) reconstruct the FO3/NV-shaped block (Skyrim SE / FO4 keep the wider 8/9-slot [`build_texture_set`](../nifblend/bridge/material_out.py)); [`export_material`](../nifblend/bridge/material_out.py) dispatches on `MaterialData.origin == "BSShaderPPLightingProperty"`. Vendoring `nifgen` from `blender_niftools_addon` / cobra-tools was attempted and abandoned (see [`docs/vendor/UPSTREAM_NIFGEN.md`](vendor/UPSTREAM_NIFGEN.md)); their codebase remains a useful design reference and may be ported under GPL-3.0, but is not a runtime dependency. License remains GPL-3.0 because the schema itself is GPL-3.0.
- **Blender target**: 5.0+ only. New extension/wheel system; numpy as wheel dep declared in manifest.
- **Game scope sequencing**: Skyrim SE first (largest modding community + BSTriShape is the tricky case), then SLE, then expand outward. Morrowind last in v1.0.
- **BSTriShape is first-class**: no silent conversion to NiTriShape — preserved through round-trip.
- **License**: **GPL-3.0-or-later** to match `nif.xml` lineage (the canonical NIF schema is GPL-3.0; any code derived from it inherits the license).

### Resolved planning decisions

1. **Starfield**: import lands in v1.1 ([Phase 9](#phase-9--starfield-import-v11)); export, morphs, cloth physics, Havok constraints stay v2. Every abstraction (GameProfile dispatch, mesh adapters, external-asset hooks) was designed with Starfield in mind so adding `format/starfield_mesh.py` was additive, not a rewrite.
2. **NIF flag storage**: typed `PropertyGroup` registered on Mesh/Material/Bone (schema-validated round-trip). Not raw ID-property dicts.
3. **Schema layer**: in-house codegen from a pinned `nif.xml` snapshot, not a vendored prebuilt nifgen. Keeps the toolchain auditable, lets us emit only the closure of types we actually need, and avoids the path-baking / generator-bug issues hit when trying to vendor cobra-tools output. Revisit if upstream `nifgen` ever stabilizes as a clean PyPI dep.

### Out of scope for v1.0

Collision (`bhk*`), Havok constraints, BSDismember editing UI, morph/shape-key controllers (`NiGeomMorpherController`), KF *export*, EGM/TRI files. Deferred to v1.1+ / v2.

**Starfield**: import lands in v1.1 (Phase 9 below). Starfield *export*, morphs (`.morph`), cloth physics (`physics_data.bin`), and Havok constraints stay v2.
